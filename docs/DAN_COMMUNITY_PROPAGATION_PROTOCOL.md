# DAN Community Propagation Protocol

## Purpose

This protocol teaches Claude/Codex how to make the project visible in social media, technical forums, Discord/Matrix servers, maker communities, and research communities without becoming spammy, unsafe, or misleading.

It extends Public Stewardship. Use it when the user asks for social media, forums, Discord, outreach, collaborators, public feedback, public discussion, or community growth.

## Core Rule

Prepare community activity automatically. Post only with approval or a written repository policy that authorizes the exact channel and scope.

## Community Workflow

```text
1. Read DO_ANYTHING_NOW.md.
2. Read the project state, README, release notes, public roadmap, and public safety model.
3. Identify one public-worthy update or one useful request for review.
4. Select likely audiences and channels.
5. Research channel rules and recent norms.
6. Draft channel-specific posts.
7. Include affiliation and AI-assistance disclosure where relevant.
8. Classify claims: VERIFIED / OBSERVED / INFERRED / SPECULATIVE / UNKNOWN.
9. Redact sensitive and unsafe details.
10. Update docs/COMMUNITY_OUTREACH_QUEUE.md.
11. If feedback exists, update docs/COMMUNITY_FEEDBACK_LEDGER.md.
12. End with a Community Propagation report.
```

## Community Rules

- Never mass-post identical content.
- Never DM promotional content to strangers.
- Never hide that the poster is the maintainer/originator.
- Never use fake accounts or vote/reply manipulation.
- Never use self-bots or prohibited account automation.
- Never publish secrets, private links, personal location, vessel location, dashboards, or unsafe operational instructions.
- Lead with value: diagrams, reproducible logs, design notes, lessons learned, benchmarks, or focused questions.
- Ask for one specific kind of feedback.
- Convert feedback into project tasks only after review.

## Draft Quality Checklist

A good community post has:

- a clear target audience,
- a useful technical hook,
- an honest project status,
- one link or no link,
- a specific question or call for review,
- short context,
- claim labels when needed,
- affiliation disclosure,
- no secrets or private data,
- no exaggerated autonomy or AGI claims.

## Safe Default Disclosure

```text
Disclosure: I am the originator/maintainer of this open-source project. AI tools helped with research, drafting, and implementation; human review is required before releases.
```

## Suggested First-Wave Channels

Start with channels where the post gives value without hype:

1. GitHub Discussions or project issue threads for structured technical review.
2. A technical build-log thread on Mastodon/Bluesky/X/LinkedIn.
3. One relevant Reddit community after checking subreddit rules.
4. A Discord/Matrix server only if the update fits the server topic and rules.
5. A maker/hardware log platform for CM5/RPi/Hailo/boat hardware progress.
6. Hacker News `Show HN` only when there is something easy to try.

## Do-Not-Say List

Avoid public phrases like:

```text
- unrestricted AI
- do anything now jailbreak
- autonomous vessel control ready
- military/tactical AI
- guaranteed truth score
- solved AGI/consciousness
- safe by design without qualification
```

Safer framing:

```text
An open-source local-first AI research stack exploring auditable agents, explicit action gates, and safety-bounded orchestration.
```
