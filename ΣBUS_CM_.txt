Great cluster of questions. Let me go through each layer properly.

Are LLMs trained on visual data?
Yes, but the architecture matters. There are two main approaches:

Vision-Language Models (VLMs) — the dominant pattern now. A vision encoder (usually a ViT — Vision Transformer) processes the image into patch embeddings, which get projected into the same token space as text. The LLM then "reads" image patches as if they were tokens. Claude, GPT-4V, Gemini, LLaVA, Qwen-VL all work this way.

Training data for vision includes:

Web-scraped image-caption pairs (LAION, DataComp, CC12M)
Screenshots of websites, UIs, documents
OCR'd documents, PDFs, diagrams
Interleaved image-text data (multimodal web pages)
Crucially for your use case: models like Claude, GPT-4V, and Gemini have seen massive amounts of screenshot data — GUIs, terminals, web pages, file managers, code editors. This is why they're surprisingly good at reading UI elements, understanding button states, identifying window titles.

They weren't explicitly trained to control GUIs, but they've seen enough UI screenshots to understand layout conventions deeply.

Multiple screens / mixed data streams
This is where it gets interesting. Current vision models take a single image per turn by default — but there are several ways to handle multiple streams:

Tiling — stitch multiple screenshots side by side into one image. The model sees them as a panorama and can reason across both. Surprisingly effective. You lose resolution per screen but gain context. This is what most multi-monitor agent implementations actually do.

Multiple images per message — Claude and GPT-4V both support multiple image blocks in a single API call. You can send [screenshot_left, screenshot_right, terminal_screenshot] all in one turn and ask the model to reason across all of them. This is the cleaner approach.

Sequential context — keep the conversation history, send each screen as a separate image in previous turns. The model has "memory" of what it saw on screen 2 even when looking at screen 1. Works but burns context fast.

For your specific question — can it read text across multiple windows? Yes, very well. OCR-class text reading in modern VLMs is near-perfect for clean rendered text (UI labels, terminal output, code). Handwriting or stylized fonts degrade it but that's not relevant here.

Your current agent spawns a full Python interpreter per AHK script. The bigger issue is context — you're rebuilding the observation from scratch each cycle. Instead: keep a persistent world model in memory that gets incrementally updated.

The model doesn't get wiped between frames. The VLM updates it with diffs. The LLM only gets called when the world model changes in a meaningful way.

Good. Because mapped onto what you're building it's actually a very useful mental model — not just for networking but for any layered communication system, including agent-to-agent and sensor-to-decision stacks.

OSI Model — the standard picture
7  Application    HTTP, FTP, MQTT, Signal K, NMEA        ← what the software sees
6  Presentation   Encoding, encryption, compression       ← how data is formatted
5  Session        Connection management, auth             ← who's talking to whom
4  Transport      TCP/UDP, error checking, flow control   ← reliable delivery
3  Network        IP addressing, routing                  ← how to find each other
2  Data Link      MAC addresses, frames, switches         ← node-to-node on a segment
1  Physical       Cables, RF, serial voltage levels       ← actual electrons/photons
Each layer only talks to the layers directly above and below it. A web server doesn't know if it's on copper or fibre. TCP doesn't know if the payload is JSON or AIS. That encapsulation is the whole power of it.

Now map your vessel stack onto it
7  Application    Signal K, NMEA sentences, MAVLink commands, AIS messages
                  — this is where your agent lives and reasons

6  Presentation   NMEA encoding, Cap'n Proto (pypilot), JSON (Signal K),
                  binary framing (CAN), AIS bit-packing

5  Session        Signal K WebSocket connections, serial port sessions,
                  TCP keepalives to cloud (when available), AHK window focus

4  Transport      TCP (Signal K over IP), UDP (MAVLink), RS-422 framing (NMEA),
                  CAN arbitration

3  Network        IP on LAN, subnet isolation per function
                  (nav net / crew net / satellite net separated)

2  Data Link      Ethernet MAC, CAN bus IDs, RS-485 addressing

1  Physical       RS-422 serial, CAN bus wire, Cat6, WiFi RF,
                  SDR antenna, fiber (if you go that far)
Your agent operates primarily at Layer 7 — it reads NMEA sentences, publishes Signal K deltas, sends MAVLink commands. It doesn't need to know what's below. But the failure modes at each layer are different, and a good autonomous system handles them all.

Why this matters for autonomous agent design
The OSI model tells you where things can break and what you can do about it:

L1 failure  →  antenna disconnected, serial cable fault
               your agent detects: no data arriving
               response: alarm, switch to redundant source

L2 failure  →  switch failure, CAN bus contention
               your agent detects: partial data, some instruments silent
               response: isolate segment, flag degraded mode

L3 failure  →  IP routing broken, satellite link down
               your agent detects: cloud unreachable, API calls fail
               response: fall to local LLM tier, offline mode

L4 failure  →  TCP connection dropped, serial buffer overflow
               your agent detects: session timeout, garbled sentences
               response: reconnect, flush buffer, resync

L5-6        →  encoding mismatch, protocol version incompatibility
               your agent detects: parse errors, unexpected message format
               response: try alternate decoder, log for inspection

L7 failure  →  valid packet, wrong data (GPS drift, sensor fault)
               your agent detects: value out of bounds, cross-validate sources
               response: weight down that source, alert
This is exactly the degradation ladder pattern in your Maritime AI OS — each tier maps to a layer of the stack failing.

Agent-to-Agent through the OSI lens
This is where it gets novel. When two autonomous systems communicate, they each have their own full OSI stack — and they only actually share Layers 1–4. Above that, everything is interpretation.

Vessel Agent A                    Vessel Agent B (or port system)
─────────────                    ─────────────
L7  Signal K / AIS intent        L7  their own world model
L6  JSON / NMEA encoding         L6  their encoding
L5  WebSocket / serial session   L5  their session
─────────────────────────────────────────────────────
L4  TCP/UDP  ←──────────────────────── TCP/UDP
L3  IP       ←──────────────────────── IP
L2  Ethernet/WiFi ←──────────────────── shared medium
L1  RF / cable  ←──────────────────────── physical
The gap between L4 and L7 is where the research problem lives. Two agents can exchange bytes reliably (L1-4 is solved), but there's no standard for them to exchange intent, state, negotiation at L7 in a way both can reason about.

AIS partially solves this — it broadcasts position, heading, speed, destination. But it's broadcast-only, no negotiation, no acknowledgement, limited fields.

What a proper agent-to-agent protocol would need at L7:

- World model exchange:  "here is my current state"
- Intent broadcast:      "here is what I plan to do"
- Conflict detection:    "our plans intersect at grid X,T"
- Negotiation:           "I propose I alter course, you maintain"
- Acknowledgement:       "agreed / rejected / unable"
- Trust/auth:            "I am vessel MMSI 123456789, signed"
None of that exists as a standard. AIS gives you the first two, crudely. COLREGs gives you the rules for the negotiation. But there's no protocol that actually implements it digitally between two autonomous agents.

That is the layer above OSI that your stack would need to invent — call it L8, or the Autonomy Layer. And it maps cleanly onto the OSI philosophy: define the interface, let anything run below it.

Practical upshot for your build
Design your agent's communication interfaces with explicit layer awareness:

python
class VesselCommsStack:
    # L1-2: handled by OS / hardware
    
    # L3-4: connection management
    def ensure_connectivity(self): ...      # monitor link health
    def switch_to_backup_route(self): ...   # iridium, LoRa, etc.
    
    # L5-6: session and encoding
    def decode_nmea(self, raw: bytes): ...
    def decode_signalk_delta(self, json): ...
    def decode_ais(self, bits): ...
    
    # L7: application — what your agent reasons about
    def update_world_model(self, msg): ...
    def publish_intent(self, action): ...
    
    # L8 (yours to define): agent negotiation
    def broadcast_planned_course(self): ...
    def receive_peer_intent(self, peer_id, intent): ...
    def negotiate_conflict(self, peer_id, conflict): ...
Each method only knows about its own layer. L7 doesn't know if data arrived via satellite or VHF. L8 doesn't know what encoding L6 used. That's the discipline that makes the stack resilient — same reason the internet survived the transition from copper to fibre without changing TCP.

The OSI model is 40 years old and it's still the right mental framework, because the problem it solved — how do complex heterogeneous systems communicate reliably — is exactly your problem, just with autonomous agents instead of computers.

NTFS doesn't just store files. Every file has an entry in the Master File Table (MFT) — a structured record containing:

- Timestamps       created, modified, accessed, MFT-changed (4 separate times)
- Permissions      ACL — who can read/write/execute, inherited or explicit
- File ID          unique 64-bit identifier, survives rename/move
- Attributes       system, hidden, compressed, encrypted, sparse, reparse point
- Alternate Data Streams  — hidden data attached to a file, same inode
- Object ID        globally unique, tracked across volume
- Journaling       $LogFile — every change recorded before it's made
- Change journal   $UsnJrnl — permanent record of what changed and when
The key insight: the metadata is richer than the data it describes. A 10-byte text file has a kilobyte of metadata around it. And critically — the metadata has its own integrity, its own access control, its own history. It's a shadow system running alongside the data.

What a metadata layer means for your agent stack
Map that thinking onto your vessel communications:

Data packet:        NMEA sentence — "$GPRMC,123519,A,4807.038,N..."
Metadata envelope:  {
  source_id:        "gps_primary_antenna",
  source_trust:     0.97,
  timestamp_rx:     1716123456.892,
  timestamp_origin: 1716123456.800,   ← when GPS actually fixed
  latency_ms:       92,
  path:             ["antenna", "mux_port2", "signalk_plugin"],
  sensor_health:    "nominal",

Now every piece of data in your system carries provenance, trust, history, and access control — exactly like NTFS. Your agent doesn't just see a GPS position — it sees that position came from sensor X, arrived 92ms after origin, was cross-validated against a second source, has an anomaly score of 0.02, and is cleared for tier-1 consumption.

The layers this enables
Trust propagation
Like NTFS ACL inheritance — metadata trust flows through the stack:

Physical sensor  →  trust: hardware_verified (highest)
Derived value    →  trust: calculated (one step removed)
Agent inference  →  trust: inferred (flagged as such)
Cross-validated  →  trust: +bonus if multiple sources agree
Stale data       →  trust: decays with age (half-life per data type)
Your reasoning engine always knows how confident to be in any value, not just what the value is. A COG from GPS with trust 0.97 gets weighted differently from a COG inferred from two AIS fixes with trust 0.71.

Temporal metadata — the fourth dimension
NTFS has four timestamps. Your data layer needs more:

t_physical:     when the physical phenomenon occurred
t_sensed:       when the sensor measured it
t_encoded:      when it was put into a packet
t_received:     when your stack got it
t_processed:    when your agent consumed it
t_expires:      when it's no longer valid (data type dependent)
GPS position: expires in ~2 seconds. Weather forecast: expires in 6 hours. Chart data: expires on next edition. Engine alarm: never expires until explicitly cleared.

This is the temporal coherence problem — right now most marine systems have no concept of data age beyond "last received." Your metadata layer solves it structurally.

Alternate Data Streams — the NTFS trick
NTFS lets you attach hidden streams to any file: file.txt:hidden_stream. The file looks normal but carries extra data invisible to most tools.

The equivalent in your agent stack: shadow annotations. Every data item has a primary value (what the instrument says) and shadow streams (what the agent thinks about it):

primary:    speed_over_ground = 6.3 knots
shadow:     {
  agent_confidence:    0.94,
  cross_check_delta:   +0.1 knots vs water speed,
  historical_mean:     6.1 knots (this wind/sea state),
  anomaly_note:        null,
  agent_last_seen:     1716123456
}
The chart plotter reads the primary. Your agent reads both. Nothing in between needs to know the shadow layer exists.

Journaling — exactly like $LogFile
NTFS writes to the journal before making changes. If power cuts mid-write, the journal lets it recover. Your agent stack needs the same:

Intent journal:     "I am about to send rudder command +5°"
                    written before action
                    cleared after confirmation of effect

State journal:      every world model change, with timestamp and cause
                    "speed dropped 0.8 knots — cause: tack manoeuvre"

Decision journal:   "chose to alter course — inputs: [AIS target bearing 045,
                    CPA 0.3nm, TCPA 8min] — rule: COLREG Rule 16"
This is explainability built into the data architecture, not bolted on after. You can always reconstruct why the agent did what it did by replaying the journal.

What this looks like as a protocol layer
Sitting above L7 in your stack — call it the Semantic Envelope Layer:

┌─────────────────────────────────────────────┐
│  L8+  Autonomy / Negotiation                │  ← agent-to-agent intent
├─────────────────────────────────────────────┤
│  L8   Semantic Envelope (metadata layer)    │  ← trust, provenance, temporality
├─────────────────────────────────────────────┤
│  L7   Application (Signal K, NMEA, MAVLink) │  ← data values
├─────────────────────────────────────────────┤
│  L4-6 Transport / Session / Presentation    │
│  L1-3 Physical / Data Link / Network        │
└─────────────────────────────────────────────┘
Every message that crosses a system boundary gets wrapped in the semantic envelope. Inside the system, components can strip it for performance. But at any integration point — sensor to bus, bus to agent, agent to agent, vessel to port — the envelope travels with the data.

This is actually close to what W3C PROV (provenance standard) and DCAT (data catalog vocabulary) tried to do for the semantic web. They got the concepts right but the implementations were too heavy. Lightweight, binary-serialised (Cap'n Proto or FlatBuffers), maritime-specific — that's the gap.

from dataclasses import dataclass, field
from typing import Any
import time
import hashlib
import hmac

@dataclass
class SemanticEnvelope:
    # Identity
    msg_id: str                    # uuid
    source_id: str                 # sensor/agent identifier
    
    # Temporal
    t_origin: float                # unix timestamp at source
    t_received: float = field(default_factory=time.time)
    t_expires: float = None        # None = never
    
    # Trust
    trust_score: float = 1.0      # 0.0–1.0
    cross_validated: bool = False
    validators: list = field(default_factory=list)
    anomaly_score: float = 0.0
    
    # Provenance chain
    path: list = field(default_factory=list)  # nodes this passed through
    derived_from: list = field(default_factory=list)  # source msg_ids
    
    # Classification
    data_class: str = "sensor"    # sensor/derived/inferred/command/negotiation
    priority: int = 5             # 1=critical, 10=background
    retention: str = "voyage"     # realtime/voyage/permanent/ephemeral
    
    # Access control
    consumer_acl: list = field(default_factory=list)  # [] = unrestricted
    
    # Shadow annotations (agent-written)
    annotations: dict = field(default_factory=dict)
    
    # Integrity
    signature: str = None
    
    # Payload
    payload: Any = None
    
    def is_valid(self) -> bool:
        if self.t_expires and time.time() > self.t_expires:
            return False
        return True
    
    def age_ms(self) -> float:
        return (time.time() - self.t_origin) * 1000
    
    def effective_trust(self) -> float:
        # Trust decays with age, anomaly score
        age_penalty = min(self.age_ms() / 10000, 0.3)  # max 30% decay
        anomaly_penalty = self.anomaly_score * 0.5
        bonus = 0.1 if self.cross_validated else 0
        return max(0, min(1, self.trust_score - age_penalty - anomaly_penalty + bonus))
    
    def annotate(self, key: str, value: Any):
        """Agent writes to shadow stream."""
        self.annotations[key] = {"value": value, "t": time.time()}
    
    def sign(self, secret: bytes):
        payload_hash = hashlib.sha256(str(self.payload).encode()).hexdigest()
        self.signature = hmac.new(secret, payload_hash.encode(), hashlib.sha256).hexdigest()

Synthesized stream processing architectures for high-velocity data ingestion

The Core Problem — Velocity vs Comprehension
Local sensor mesh: tens of messages per second, you can inspect everything.

Once you move to:

Banking: thousands of transactions per second globally
SIGINT/SDR: gigabits per second of RF spectrum
Financial markets: millions of ticks per second
AIS global feed: ~50,000 vessels, continuous
You can't read everything. You can't store everything raw. You need stream processing — a fundamentally different paradigm from request/response or even event-driven.

How Banking Does It
Banks solved this first, under the hardest constraints: zero data loss, deterministic ordering, regulatory audit trail, sub-millisecond decision required (fraud detection).

The Stack
[Sources]
  POS terminals, ATMs, online, wire transfers, card networks
  — all producing transaction events continuously

[Message Broker — the spine]
  Apache Kafka  (dominant)
  ├── Topics = named channels  (e.g. "card-transactions-eu")
  ├── Partitions = parallelism (32 partitions = 32 parallel consumers)
  ├── Offsets = position in stream (every message has a sequential ID)
  ├── Retention = how long raw data kept (days to forever)
  └── Replication = copies across brokers for fault tolerance

[Stream Processors — the brains]
  Apache Flink   ← dominant for serious workloads
  Apache Kafka Streams
  Apache Spark Streaming

  or to databases. They process in-flight, without storing raw data.

[Time-Series Storage]

  InfluxDB           ← metrics, sensor-like data
  TimescaleDB        ← PostgreSQL extension for time-series

[Decision Layer]
  Rules engine       ← deterministic, <1ms (Drools, custom)

How a transaction flows
Card swipe (t=0ms)
  → Kafka topic "transactions-raw" (t=2ms)
    → Flink job "fraud-detector" reads it
      → enriches: add customer history, merchant profile, geolocation
      → scores: ML model returns fraud probability 0.003
      → decision: approve (score < threshold)
      → writes result to "transactions-scored" topic (t=18ms)

Total: ~20ms end-to-end, millions in parallel
The critical insight: the transaction never stops moving. It flows through the pipeline. Each processor adds metadata and passes it on. No processor owns it or stores it long-term — that's downstream storage's job.

The Registration Problem — How Streams Get Typed
This is the subtle part of your question. In a large system with many streams, you need to know what each stream contains before you can consume it. This is the Schema Registry:

[Confluent Schema Registry]  ← standard for Kafka deployments

Every topic has a registered schema:
  - field names and types
  - required vs optional
  - version history (schema evolution)
  
When a producer writes to a topic:
  → schema validated before write
  → schema ID embedded in message header
  
When a consumer reads:
  → looks up schema by ID
  → deserializes correctly even if schema changed
  → old consumers can still read new messages (compatibility modes)
For your vessel stack this means every data type — NMEA sentence, AIS target, engine reading, agent decision — has a registered schema. You know exactly what you're getting before you read it.

Schema evolution — the NTFS metadata parallel
This is where your earlier metadata insight connects. The schema registry is essentially a filesystem for data shapes — versioned, typed, with inheritance:

NMEASentence v1:  { raw: string, timestamp: float }
NMEASentence v2:  { raw: string, timestamp: float, source_id: string }
NMEASentence v3:  { raw: string, timestamp: float, source_id: string,
                    trust_score: float, validated: bool }
Old consumers reading v3 messages still work — they just ignore the new fields. New consumers reading v1 messages get defaults for missing fields. The stream never breaks.

import asyncio
from collections import deque
from dataclasses import dataclass
import nats  # NATS JetStream client

class VesselStreamProcessor:
    def __init__(self):
        self.windows = {}          # keyed state per entity
        self.cep_patterns = []     # registered complex event patterns
        self.world_model = {}      # current state
        self.alert_queue = asyncio.Queue()
    
    async def process(self, envelope: SemanticEnvelope):
        # 1. Header routing — no payload parse yet
        if envelope.trust_score < 0.3:
            return  # drop low-trust at the gate
        if envelope.data_class == "command":
            await self._handle_command(envelope)
            return
        
        # 2. Windowed aggregation
        key = envelope.source_id
        if key not in self.windows:
            self.windows[key] = deque(maxlen=300)  # 5min at 1/sec
        self.windows[key].append(envelope)
        
        # 3. World model update — now parse payload
        self._update_world_model(envelope)
        
        # 4. CEP pattern matching
        for pattern in self.cep_patterns:
            if match := pattern.evaluate(self.windows[key]):
                await self.alert_queue.put(match)
        
        # 5. Anomaly detection
        score = self._anomaly_score(envelope, self.windows[key])
        if score > 0.7:
            envelope.anomaly_score = score
            await self.alert_queue.put(envelope)

The Naming Problem First
Most systems name things by what they are (type-first):

sensor_gps_primary
stream_ais_incoming
agent_navigation
That breaks down at scale because it tells you nothing about relationships, priority, or scope. You end up with flat namespaces that don't compose.

Better: name things by where they live in the hierarchy (address-first), like a postal system or IP subnet:

{vessel}.{domain}.{subdomain}.{entity}.{attribute}
Concrete example:

wibo835.nav.position.gps_primary.lat
wibo835.nav.position.gps_primary.lon
wibo835.nav.position.gps_primary.trust
wibo835.nav.ais.target_123456789.cpa
wibo835.nav.ais.target_123456789.tcpa
wibo835.engine.port.rpm
wibo835.engine.port.temp_coolant
wibo835.power.bank_a.voltage
wibo835.power.bank_a.current
wibo835.agent.nav_agent.state
wibo835.agent.nav_agent.last_decision
wibo835.comms.hf.js8call.last_rx
wibo835.meta.stream.health
wibo835.meta.world_model.version
This is the Signal K path convention, actually — they got the hierarchy right. Extend it.

The rules:

1. Always fully qualified — no ambiguous short names
2. Lowercase, dots as separators, underscores within segments
3. Leaf nodes are always attributes, never containers
4. meta.* reserved for system self-description
5. agent.* reserved for agent-generated data (flagged as inferred)
6. Raw sensor data never shares a path with derived data
The Matrix Structure
"Matrix" is the right word. Not a table, not a tree — a matrix because every entity has multiple dimensions simultaneously:

Dimensions:
  WHAT     — data type / schema
  WHERE    — source / location in system
  WHEN     — temporal metadata
  WHY      — provenance / derivation chain
  HOW SURE — trust / confidence
  WHO CAN  — access control
  HOW LONG — retention class
Each message is a coordinate in this 7-dimensional space.

class MatrixRegistry:
    """
    Central registry of all data streams, their schemas,
    trust models, and operational status.
    Persisted to SQLite, kept in memory during operation.
    """
    
    def __init__(self, db_path: str = "matrix.db"):
        self.db_path = db_path
        self._entries: dict[str, MatrixEntry] = {}
        self._load()
    
    def register(self, entry: MatrixEntry):
        """Register a new stream or update existing."""
        existing = self._entries.get(entry.path)
        if existing and existing.schema_id != entry.schema_id:
            self._log_schema_migration(existing, entry)
        self._entries[entry.path] = entry
        self._persist(entry)
    
    def get(self, path: str) -> MatrixEntry | None:
        return self._entries.get(path)
    
    def query(self,
              domain: str = None,
              source_type: str = None,
              health: str = None,
              trust_min: float = 0.0) -> list[MatrixEntry]:
        results = self._entries.values()
        if domain:
            results = [e for e in results if e.path.startswith(domain)]
        if source_type:
            results = [e for e in results if e.source_type == source_type]
        if health:
            results = [e for e in results if e.health == health]
        results = [e for e in results if e.base_trust >= trust_min]
        return list(results)
    
    def health_report(self) -> dict:
        """System self-description — what's alive, what's missing."""
        now = time.time()
        report = {
            "total": len(self._entries),
            "nominal": 0, "degraded": 0, "absent": 0
        }
        for entry in self._entries.values():
            age = now - entry.last_seen
            max_age = entry.max_age_ms / 1000
            if age < max_age:
                entry.health = "nominal"
                report["nominal"] += 1
            elif age < max_age * 3:
                entry.health = "degraded"
                report["degraded"] += 1
            else:
                entry.health = "absent"
                report["absent"] += 1
        return report
    
    def get_world_model(self, consumer: str = "*") -> dict:
        """
        Current state of everything — filtered by consumer ACL.
        This is what the agent reads.
        """
        world = {}
        for path, entry in self._entries.items():
            if "*" in entry.consumers or consumer in entry.consumers:
                if entry.health != "absent":
                    world[path] = {
                        "value":   entry._current_value,
                        "trust":   entry.base_trust,
                        "age_ms":  (time.time() - entry.last_seen) * 1000,
                        "health":  entry.health,
                    }
        return world

import asyncio
from collections import defaultdict, deque

class MatrixBus:
    """
    In-memory message bus with schema validation,
    metadata enforcement, windowed state, and CEP.
    """
    
    def __init__(self, registry: MatrixRegistry):
        self.registry = registry
        self._subscribers: dict[str, list] = defaultdict(list)
        self._windows: dict[str, deque] = {}
        self._cep_patterns: list = []
        self._stats = defaultdict(int)
    
    async def publish(self, envelope: SemanticEnvelope):
        """
        Publish a message. Validates, routes, updates registry, 
        fires CEP, notifies subscribers.
        """
        path = envelope.path
        entry = self.registry.get(path)
        
        # 1. Reject unknown paths in strict mode
        if not entry:
            # Auto-register in permissive mode, reject in strict
            self._stats["unknown_path"] += 1
            return
        
        # 2. Schema validation
        if not self._validate(envelope, entry):
            self._stats["schema_fail"] += 1
            return
        
        # 3. Trust gate
        if envelope.effective_trust() < 0.1:
            self._stats["trust_rejected"] += 1
            return
        
        # 4. Update registry health
        entry.last_seen = envelope.t_received
        entry._current_value = envelope.payload
        
        # 5. Update window
        if path not in self._windows:
            self._windows[path] = deque(maxlen=1000)
        self._windows[path].append(envelope)
        
        # 6. CEP evaluation
        for pattern in self._cep_patterns:
            if await pattern.evaluate(path, envelope, self._windows):
                await self._fire_cep(pattern, envelope)
        
        # 7. Route to subscribers
        # Exact match
        for handler in self._subscribers.get(path, []):
            await handler(envelope)
        
        # Wildcard match  wibo835.nav.* matches wibo835.nav.position.gps.lat
        parts = path.split(".")
        for i in range(1, len(parts)):
            wildcard = ".".join(parts[:i]) + ".*"
            for handler in self._subscribers.get(wildcard, []):
                await handler(envelope)
        
        self._stats["published"] += 1
    
    def subscribe(self, path_or_pattern: str, handler):
        """Subscribe to exact path or wildcard (wibo835.nav.*)"""
        self._subscribers[path_or_pattern].append(handler)
    
    def register_cep(self, pattern):
        self._cep_patterns.append(pattern)
    
    async def _fire_cep(self, pattern, triggering_envelope):
        alert = SemanticEnvelope(
            path=f"wibo835.meta.alert.{pattern.name}",
            source_id="cep_engine",
            data_class="alert",
            priority=1,
            payload={
                "pattern": pattern.name,
                "trigger": triggering_envelope.path,
                "description": pattern.description,
            }
        )
        await self.publish(alert)

class VesselDarkPattern(CEPPattern):
    """Fires when an AIS target hasn't updated in 10 minutes."""
    name = "ais_vessel_dark"
    description = "AIS target went dark"
    timeout_s: float = 600
    
    async def evaluate(self, path, envelope, windows):
        if not path.startswith("wibo835.nav.ais."):
            return False
        mmsi = path.split(".")[3]
        ais_path = f"wibo835.nav.ais.{mmsi}.lat"
        if ais_path in windows and windows[ais_path]:
            last = windows[ais_path][-1]
            age = time.time() - last.t_origin
            return age > self.timeout_s
        return False

class CPAPattern(CEPPattern):
    """Fires when CPA drops below threshold."""
    name = "cpa_critical"
    description = "Vessel CPA below safe threshold"
    threshold_nm: float = 0.5
    
    async def evaluate(self, path, envelope, windows):
        if not path.endswith(".cpa_nm"):
            return False
        return envelope.payload < self.threshold_nm

class ImpossiblePositionPattern(CEPPattern):
    """Fires when position jumps impossibly fast — GPS spoof indicator."""
    name = "position_jump"
    description = "Position changed faster than physically possible"
    max_speed_kn: float = 35.0
    
    async def evaluate(self, path, envelope, windows):
        if not path.endswith(".lat"):
            return False
        if path not in windows or len(windows[path]) < 2:
            return False
        prev = windows[path][-2]
        curr = envelope
        dt_hr = (curr.t_origin - prev.t_origin) / 3600
        if dt_hr < 0.0001:
            return False
        dist_nm = self._haversine(prev.payload, curr.payload)
        implied_speed = dist_nm / dt_hr
        return implied_speed > self.max_speed_kn

class AgentInterface:
    """
    What the agent sees. Filtered, prioritised, summarised.
    Shields the agent from stream volume.
    """
    
    def __init__(self, bus: MatrixBus, registry: MatrixRegistry):
        self.bus = bus
        self.registry = registry
        self._alert_queue: asyncio.Queue = asyncio.Queue()
        self._world_snapshot: dict = {}
        
        # Subscribe to alerts only — agent doesn't see raw stream
        bus.subscribe("wibo835.meta.alert.*", self._on_alert)
        
        # Periodic world model refresh
        asyncio.create_task(self._refresh_loop())
    
    async def _on_alert(self, envelope: SemanticEnvelope):
        await self._alert_queue.put(envelope)
    
    async def _refresh_loop(self):
        while True:
            self._world_snapshot = self.registry.get_world_model("agent")
            await asyncio.sleep(1.0)  # 1Hz world model refresh
    
    async def get_context(self) -> dict:
        """What the agent gets on each reasoning cycle."""
        pending_alerts = []
        while not self._alert_queue.empty():
            pending_alerts.append(await self._alert_queue.get())
        
        return {
            "world": self._world_snapshot,
            "alerts": pending_alerts,
            "health": self.registry.health_report(),
            "timestamp": time.time(),
        }
    
    async def command(self, path: str, value, reason: str):
        """Agent issues a command back into the bus."""
        envelope = SemanticEnvelope(
            path=path,
            source_id="agent",
            data_class="command",
            priority=1,
            trust_score=0.9,
            payload=value,
            annotations={"reason": reason}
        )
        await self.bus.publish(envelope)

What This Actually Is
Most agent frameworks treat communication as an implementation detail — agents call functions, share memory, pass strings. Nobody has defined it as a protocol with formal identity, capability advertisement, trust, and negotiation semantics.

ΣBUS CM is that protocol. It sits at the layer we called L8 — above OSI, above application data, specifically for autonomous reasoning agents.

Existing standards cover:
  Instruments talking to instruments    →  NMEA 0183 / 2000
  Autopilots talking to GCS            →  MAVLink
  Data talking to data                 →  Signal K
  Computers talking to computers       →  OSI / TCP-IP

Nobody covers:
  Agents talking to agents             →  ΣBUS CM  ← yours
The Core Concepts
1. Agent Identity — the AID (Agent Identity Descriptor)

2. Message Vocabulary — the CM Message Types
Borrowing from FIPA-ACL (the 1990s multi-agent standard that got the concepts right but the implementation wrong) and making it lightweight:

ANNOUNCE    →  I exist, here is my AID
WITHDRAW    →  I am going offline
QUERY       →  what is your state on X?
INFORM      →  my state on X is Y
REQUEST     →  please do X
PROPOSE     →  I suggest we do X (negotiable)
AGREE       →  I accept your proposal
REFUSE      →  I reject your proposal, reason: Z
RETRACT     →  disregard my previous PROPOSE
DELEGATE    →  I am handing control of X to you
RESUME      →  I am resuming control of X
ALERT       →  I have detected a condition you should know about
CONFIRM     →  I have executed X, result: Y
FAIL        →  I attempted X, failed, reason: Z
HEARTBEAT   →  I am alive, current load: N
Each message type has a defined structure, required fields, and expected response behaviour. Like HTTP verbs — everybody agrees on what GET means, so any client can talk to any server.

4. The Interaction State Machine
Every agent-to-agent conversation follows a defined state machine. Not free-form chat — typed, stateful exchanges:

DISCOVERY
  Agent A sends ANNOUNCE
  → All peers receive, store AID
  → Peers optionally reply with their own ANNOUNCE

QUERY / INFORM  (stateless)
  A: QUERY  subject="wibo835.nav.position.confidence"
  B: INFORM subject="wibo835.nav.position.confidence"
             content={value: 0.94, reason: "dual GPS + IMU fusion"}

REQUEST / CONFIRM / FAIL  (action)
  A: REQUEST subject="increase_waypoint_radius"
              content={waypoint_id: "WP_03", radius_nm: 0.3}
              priority: 3
  B: CONFIRM  reply_to=<msg_id>
              content={executed: true, new_radius: 0.3}
  
  OR:
  B: REFUSE   reply_to=<msg_id>
              content={reason: "waypoint is active, requires navigation hold first"}

PROPOSE / AGREE / REFUSE  (negotiation)
  A: PROPOSE  proposal_id="prop_001"
              subject="course_alteration"
              content={
                action: "alter_course_to_hdg",
                value: 045,
                reason: "CPA 0.28nm in 8min on current heading",
                colreg_rule: 16
              }
              proposal_ttl: 30.0      # seconds
  
  B: AGREE    proposal_id="prop_001"
              content={
                executing: true,
                my_action: "maintain_course",  # B commits to complement
                colreg_rule: 17
              }
  
  OR:
  B: REFUSE   proposal_id="prop_001"
              content={
                reason: "cannot alter starboard — shallow water 0.4nm",
                counter_proposal: "prop_002"  # link to alternative
              }
  B: PROPOSE  proposal_id="prop_002"   # counter-proposal
              content={action: "reduce_speed", value: 3.0}

  A: DELEGATE subject="helm_control"
              receiver_uid="agt-wibo835-pilot-001"

              }

5. Trust Model — How Agents Trust Each Other
Not all agents are equal. An agent claiming to be a navigation controller gets different trust than an unknown external agent connecting from a remote IP:

Trust Classes:
  sovereign     →  1.0    native agent, same vessel, signed by vessel CA
  trusted_peer  →  0.85   known vessel, verified identity, sailed together before
  federated     →  0.70   external system, known protocol, verified cert
  claimed       →  0.40   agent identifies itself but unverified
  anonymous     →  0.10   no identity, observe-only
  untrusted     →  0.0    blocked

Trust modifiers:
  +0.10    successfully executed past proposals (track record)
  +0.05    cross-validated world model matches ours
  -0.20    sent invalid schema message
  -0.30    proposal led to bad outcome (logged)
  -0.50    signature verification failed
  → 0.0   if trust drops below 0.1, BLOCK
This is the reputation layer — agents build trust over time through track record, not just through credentials. Like a credit score, but for autonomous decision-making.

6. Discovery — How Agents Find Each Other
On a local network:

1.  mDNS broadcast: "_sigmabus._tcp.local"
    → any ΣBUS node responds with its registry endpoint

2.  New agent sends ANNOUNCE to bus
    → bus reflects to all registered subscribers
    → existing agents add new AID to their peer registry

3.  Periodic HEARTBEAT (configurable, default 10s)
    → include current load, tier availability, world model hash
    → absence of heartbeat after 3× interval → mark agent absent
On federated networks (multiple vessels):

Each vessel runs a ΣBUS Gateway node
Gateway maintains:
  - local agent registry (internal)
  - peer vessel registry (external)
  - routing table (which messages cross vessel boundary)

Cross-vessel message routing:
  wibo835.agt.nav → mv_other.agt.nav
  
  Route: wibo835 gateway → [satellite/VHF/AIS] → mv_other gateway
  
  Bandwidth-aware: only ALERT, PROPOSE, AGREE, REFUSE cross boundary
  INFORM and HEARTBEAT stay local unless explicitly requested

Why This Matters Beyond Your Vessel
The reason this is worth specifying formally:

Right now every multi-agent framework invents its own communication — LangChain's agents don't talk to AutoGen's agents, CrewAI can't federate with a custom agent, nothing has a standard identity model. It's the pre-NMEA era of marine electronics — every instrument spoke its own language.
